# **1\. Executive Summary**

H-SETS Digital Ecosystem Platform is a unified, enterprise-grade digital presence that consolidates seven interconnected business units — Corporate Website, Academy, Community, Consulting, AI Solutions, Resource Hub, and Career Programs — into a single integrated platform. The platform is designed to drive measurable business outcomes across three commercial dimensions: B2C talent development, B2B corporate services, and long-term brand authority.

## **1.1 Strategic Intent**

The current H-SETS digital footprint operates as a collection of disconnected touchpoints. This platform initiative replaces that fragmented experience with a cohesive ecosystem in which every user interaction — whether a student enrolling in a cohort, a business owner booking a consultation, or a corporate client requesting a custom training programme — is part of a unified, data-connected journey that serves both the user and H-SETS' revenue objectives.

## **1.2 Platform Pillars**

| Pillar | Purpose | Primary Audience | Revenue Impact |
| :---- | :---- | :---- | :---- |
| Corporate Growth | Generate qualified leads for H-SETS Digital & IT Solutions | Business owners, CTOs, executives | Direct — consulting & project revenue |
| Education | Deliver structured cohort-based training via H-SETS Academy | Students, career-switchers, professionals | Direct — tuition fees & corporate training |
| Talent Development | Convert graduates into employable, hireable professionals | Graduates, employers, HR teams | Indirect — reputation & partner fees |
| Community & Authority | Build a technology ecosystem and thought leadership brand | All audiences | Long-term — retention & referrals |

## **1.3 Success Definition**
This platform will be considered successful when it demonstrably:

* Reduces cost-per-lead by at least 40% within 18 months of launch through organic SEO and content authority.

* Achieves a minimum 70% cohort fill rate across all active academy programmes.

* Maintains a Net Promoter Score (NPS) of 50+ across student and client segments.

* Achieves Top-3 Google ranking for at least 15 target commercial keywords within 12 months.

* Generates AI-cited results (ChatGPT, Perplexity, Gemini) for at least 10 branded queries within 12 months.

# **6\. Academy Module**

## **6.1 Module Overview**

The Academy is a cohort-based, instructor-led learning management system embedded within the main platform. It is not a standalone LMS — it shares the authentication layer, CRM, and user profile system with the rest of the platform. The Academy must support both public-facing discovery (course catalogue, cohort listings) and authenticated student experiences (dashboard, assignments, certificates).

## **6.3 Cohort-Based Learning System**

### **6.3.1 Cohort Configuration (Admin)**

Administrators can create and manage cohorts via the Admin Super Dashboard. Each cohort is a discrete, time-bound instance of a programme.

| Field | Type | Validation Rules |
| :---- | :---- | :---- |
| Cohort Name | Text | Required. Max 100 characters. Auto-suggested as \[Programme\] — Cohort \[N\] |
| Programme | Dropdown (relation) | Required. Must link to an active programme record |
| Start Date | Date picker | Required. Must be minimum 14 days from creation date |
| End Date | Date picker | Required. Must be after start date |
| Capacity | Number | Required. Min: 5\. Max: 200\. Default: 30 |
| Fee (Full) | Currency (NGN) | Required. Must be \> 0 |
| Fee (Installment 1\) | Currency (NGN) | Optional. Must be \< Full Fee |
| Fee (Installment 2\) | Currency (NGN) | Optional. Auto-calculated as Full Fee minus Installment 1 |
| Scholarship Slots | Number | Optional. Max \= Capacity |
| Assigned Instructors | Multi-select (relation) | Required. Min 1 instructor |
| Status | Enum | Draft / Open / Full / In Progress / Completed / Archived |
| Application Deadline | Date picker | Optional. Defaults to 3 days before start date |

### **6.3.2 Cohort Landing Page**

Each cohort has a publicly-accessible landing page at /academy/cohorts/\[id\]. It must be optimised for conversion and SEO.

* Programme overview and learning outcomes (bulleted, benefits-led)

* Full week-by-week curriculum with module titles and topics

* Duration, schedule format (full-time / part-time / weekend), weekly hours

* Payment information: full fee, installment breakdown, scholarship availability

* Instructor profile card: name, photo, title, brief bio, LinkedIn link

* Available seats counter (real-time, from database)

* Social proof: 1–3 testimonials from previous cohort graduates

* FAQs specific to the programme

* Primary CTA: Apply Now — triggers enrollment workflow

* Secondary CTA: Join Waitlist — if cohort is full

### **6.3.3 Enrollment Workflow**

The enrollment process must be frictionless, clearly progressive, and support manual admin review before payment is required.

| Step | Actor | System Action | Outcome |
| :---- | :---- | :---- | :---- |
| 1 — Application | Prospective Student | Completes application form (name, email, phone, motivation, background). Form validates required fields. | Application record created in CRM with status: Pending Review |
| 2 — Admin Review | Academy Admin | Admin reviews application in dashboard. Can Approve, Reject, or Request More Info. | Applicant receives automated email notification with outcome |
| 3 — Acceptance | System (automated) | On approval: system sends acceptance email with payment link and deadline (72 hours to pay or seat released). | Payment link generated. Seat reserved temporarily. |
| 4 — Payment | Student | Pays via Paystack or Flutterwave. Selects full or installment plan. System records transaction. | Payment confirmed. Enrollment record created. |
| 5 — Portal Access | System (automated) | Student account created or updated. Cohort enrolled. Welcome email sent. Student appears in instructor dashboard. | Student can access Student Portal. |

### **6.3.4 Waitlist System**

| Trigger | Automatic — activates when cohort capacity is reached |
| :---- | :---- |
| **Entry** | Visitor clicks Join Waitlist and submits name \+ email (no full application required) |
| **Queue Logic** | FIFO — first on waitlist gets first offer when a seat opens |
| **Notification** | System emails waitlist position confirmation on entry. Automated email \+ SMS when a seat becomes available (24-hour response window) |
| **Admin Control** | Admin can manually promote from waitlist or clear waitlist entries from dashboard |

The Learning Center is the primary content delivery interface. It is structured by module, then by lesson.

| Module Navigation | Left sidebar: collapsible module list with completion status icons (not started, in progress, completed) |
| :---- | :---- |
| **Content Types** | Video lessons (hosted on Cloudflare Stream or embedded Loom/Vimeo), PDF slides, written notes, downloadable resources |

| **Downloads** | Lesson-level downloadable assets (slide decks, code starters, templates) with download tracking. |

| Feature | Specification |
| :---- | :---- |
| Assignment Types | Written submission (text editor), File upload (PDF, DOCX, ZIP — max 25 MB), URL submission (for live project links), Presentation upload (PPTX, PDF) |

## **7.4 Attendance & Assessment**

| Live Session Tracking | Instructors trigger attendance check within first 15 minutes of live session. Students who click Confirm Present within a 5-minute window are marked attended. |
| :---- | :---- |
| **Attendance Threshold** | Minimum 70% attendance required for certificate eligibility. Students below 50% receive a warning notification. |
| **Assessment Engine** | Quizzes: MCQ, true/false, short answer (auto-graded). Mid-cohort Exam: configurable. Practical Projects: manual grading by instructor. |
| **Grading Scale** | 0–100. Minimum pass score configurable per programme (default: 50). Distinction: 75+. Merit: 60–74. Pass: 50–59. |
| **Retake Policy** | Configurable per assessment: 0, 1, or unlimited retakes. Retake scores recorded separately; highest score is used for certification. |

# **8\. Certification System**

## **8.1 Certificate Eligibility Rules**

Certificate generation is automated and triggered only when all three conditions are simultaneously met. The system checks eligibility nightly and on each relevant event (assignment graded, session attended).

| Requirement | Threshold | Override Allowed? |
| :---- | :---- | :---- |
| Attendance | ≥70% of scheduled live sessions attended | Yes — Admin can grant attendance waiver |
| Assignment Completion | 100% of required assignments submitted (not necessarily passed) | Yes — Admin can mark assignment exempt |
| Assessment Score | Minimum pass score achieved on final assessment (default 50/100) | Yes — Admin override with reason logged |

## **8.2 Certificate Design & Content**

| Format | A4 landscape PDF, generated programmatically from a branded template |
| :---- | :---- |
| **Fields** | Student full name (as entered during registration), Programme name, Cohort name and dates, Certificate ID (8-character alphanumeric, unique), Issue date, QR code linking to verification page, H-SETS logo and authorised signature |
| **Delivery** | Emailed to student automatically upon eligibility. Available for download in Student Portal under Certificates tab. |
| **Verification** | Public page at /verify/\[certificate-id\] confirms authenticity. Displays: name, programme, cohort, issue date. No grade information shown publicly. |
| **Revocation** | Admin can revoke certificate (e.g., integrity violation). Revoked certificates return Not Valid on verification page. Student notified by email. |

## **21.3 Security Requirements**

* All traffic HTTPS only. HSTS enabled via Cloudflare.

* Supabase Row-Level Security (RLS) policies enforced on ALL database tables. No bypass allowed.

* API rate limiting on all public endpoints: 100 requests/minute per IP. 429 response beyond limit.

* File upload validation: server-side MIME type check, file size limit, antivirus scan (ClamAV integration in Phase 2).

* Payment data: never stored on H-SETS servers. Paystack/Flutterwave handle all card data (PCI-DSS compliant).

* Admin actions logged with user ID, timestamp, and action type. Logs retained 90 days.

* Password policy: minimum 8 characters, 1 uppercase, 1 number. bcrypt hashing via Supabase Auth.

* Certificate PDF generation server-side only — no client-side manipulation possible.
